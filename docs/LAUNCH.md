# Open-Source Launch Plan

## Objective

Build a credible global research project that earns attention through transparent equations, privacy-preserving local use, and high-quality public discussion.

A target such as 40,000 GitHub stars in one week is an exceptional distribution outcome, not an engineering acceptance criterion. The controllable objective is to maximize:

- Qualified repository visits
- Live-demo completion
- README-to-star conversion
- Meaningful model issues and pull requests
- Citations and discussion by researchers, operators, and finance leaders

## Launch prerequisites

- Public GitHub repository with MIT license
- Passing CI
- GitHub Pages live demo
- English README and model specification
- English, Traditional Chinese, Japanese, French, and Spanish interface support
- Traditional Chinese companion documentation
- Local browser-generated PDF research reports
- Contribution and security policies
- Sanitized example scenarios
- One concise product video or animated walkthrough
- Three static screenshots: inputs, baseline attribution, uncertainty range
- A tagged release with a clear changelog

## Positioning

Primary message:

> Most AI ROI calculators assume productivity. This one lets review, rework, weak demand, low margins, and pre-existing financial pressure make the answer negative.

Supporting messages:

- Runs locally; company data is not uploaded
- Every equation is inspectable
- Separates no-AI baseline pressure from AI-attributed impact
- Accepts negative productivity
- Designed for research and internal scenario testing, not workforce recommendations

## One-week distribution plan

### Before launch

- Recruit 10–20 reviewers across finance, engineering, HR research, AI operations, and economics.
- Resolve terminology and equation disputes before public release.
- Prepare three reproducible scenarios: positive, negative, and misleading-single-metric.
- Publish a short model methodology article linking directly to the repository and live demo.

### Launch day

- Publish the repository, GitHub Pages demo, and first tagged release together.
- Post one technical explanation focused on baseline attribution and negative productivity.
- Share separate operator-focused and researcher-focused summaries instead of one generic announcement.
- Ask reviewers to open substantive issues rather than posting only endorsements.

### Days 2–7

- Respond to issues within one working day.
- Convert recurring questions into README and model-spec improvements.
- Publish one scenario analysis per day using sanitized or public data.
- Highlight accepted external contributions.
- Track which channels produce contributors rather than only traffic.

## Repository conversion checklist

- The first README viewport explains the problem, local privacy, and live-demo path.
- Equations are visible without reading source code.
- The difference between public data and synthetic assumptions is explicit.
- `npm test` works without dependency installation.
- Issue templates make model debate structured.
- Pull requests have a clear acceptance checklist.
- A new contributor can find model logic in `calculator.js` and documentation in `docs/MODEL.md`.

## Metrics

Track daily:

- Unique repository visitors
- GitHub Pages visitors
- Stars per unique repository visitor
- Forks
- New issues
- Model-change proposals
- External pull requests
- Contributors with merged work
- Repeat visitors

Do not optimize only for stars. A smaller group of credible contributors can improve long-term adoption more than low-intent traffic.
